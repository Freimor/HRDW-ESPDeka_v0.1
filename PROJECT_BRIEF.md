# Project Brief: ESPDeka philosophy

## 1. Mission (Миссия)
Разработать прошивку для устройства offgrid коммуникации на базе jc4880p443. Устройство имеет GNSS модуль для навигации и LORA 868mHz приемо-передатчик для обмена информацией (сообщения, голос, картинки, телеметрия) с другими устройствами на базе сети LORA, по аналогии с meshtastic и meshcore.

Устройство предполагается использовать в групповых путешествиях для обеспечения связи и навигации в районах, где может отсутствовать качественная мобильная связь. В лучшем варианте устройство должно выполнять теже функции что и CivTAK/ATAK. Пользователь будет иметь доступ к карте местности, возможностью отмечать на ней события, координаты для других пользователей в общей группе, полный функционал месенджера с возможностью транслировать свою телеметрию, связанных с помощью приемо-передатчика LORA.

UI приложения разрабатывается в Figma. 

Возможные пользовательские сценарии:
1. Пользователь, находясь в одной группе контактов (канал) LORA, обменивается с другими контактами своими координатами, сообщениями (текстовые, голосовые, визуальные), телеметрией с датчиков своего устройства.
1. Пользователь может подключиться к устройству через Bluetooth для использования LORA на телефоне/ноутбуке, для обновления прошивки устройства.

## 2. Development roadmap
1. Standalone device (without Bluetooth connectivity)
    1.1 Поднимаем RTOS, пробуем написать базовое приложение - чат. UI придумывает Cursor AI. 
    Критерий успеха - приложение запустилось на устройстве. Есть возможность написать сообщение виртуальному адресату. Выполняется логирование сообщений.
    1.2 Пробуем подключиться к GNSS, его компасу, передатчику LORA. Пусть пишет получаемые данные в чат. 
    Критерий успеха - в чат, после отправки команды /start от виртуального контакта приходят сообщения, с периодичностью 1 секунда, с показаниями GNSS, LORA. При отправке команды /stop контакт перестает присылать сообщения.
    1.3 Делаем интеграцию с сетью Meshtastic - устройство должно уметь отправлять текстовые сообщение определенному клиенту сети Meshtastic. Также есть меню настройки LORA, чтобы прописать параметры сети, как это реализовано в Meshtastic.
    Критерий успеха - в приложении, в пункте меню "Настройка Meshtastic", можно прописать настройки сети. Есть возможность добавить пользователя сети Meshtatic в контакты и написать ему. От контакта приходят сообщения.

## 3. Core Philosophy (Философия дизайна)
- **Safety & Determinism:** Код должен легко читаемым для разработчика, нужны достаточные, но не избыточные комментарии к функциям. Никаких "зависаний", утечек памяти или неопределенного поведения.
- **SuperApp:** Приложение "комбайн" - чат, карта местности, меню специальных функций.

## 4. Target Device & Architecture
- **Core:** jc4880p443 (ESP32-P4 core + ESP32-C6 for WiFi and Bluetooth communication).
- **Expansion:** "Expand IO" connector — единственный интерфейс для подключения внешних модулей.
- **Framework:** ESP-IDF + FreeRTOS.
- **Language:** C/C++ (MISRA C:2012/2023, BARR-C compliant).

Помимо базовой конфигурации устройства jc4880p443 к нему будут также подключены:

LR1121 868mHz LORA приемник-передатчик (на плате находится ESP32-C3):
TX - IO52
RX - IO51

GNSS RUSHFPV M10 PRO:
SCL - IO8
SDA - IO7
TX - IO31
RX - IO29

Дополнительные датчики будут подключаться по I2C:
SCL - IO8
SDA - IO7

## 5. Key Features (Ключевые функции)
1. **Stable Core:** Базовая инициализация jc4880p443, управление питанием, watchdog, безопасное восстановление после сбоев.
2. **Expand IO HAL:** Абстрактный слой, который предоставляет унифицированный API для работы с любыми устройствами на разъеме (I2C, SPI, UART, GPIO).
3. **Dynamic Driver Loading (Optional):** Возможность добавлять новые драйверы без перекомпиляции ядра (если применимо).
4. **Telemetry & Logging:** Структурированное логирование состояния системы и подключенных устройств.
5. **OTA Updates:** Безопасное обновление firmware "по воздуху" с проверкой целостности.

## 6. Non-Goals (Чего мы НЕ делаем)
- Мы НЕ используем динамическое выделение памяти (malloc) в прикладном коде.
- Мы НЕ пишем код, который зависит от конкретных пинов Expand IO в бизнес-логике. Только через HAL.

## 7. Success Criteria (Критерии успеха)
- Код проходит статический анализ (MISRA checker) без критических ошибок.
- Потребление памяти в RAM стабильно и не растет со временем.
